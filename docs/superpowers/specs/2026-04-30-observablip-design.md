# observablip — design

A read-only Claude Code skill that audits a target codebase for poor
observability practices, missing telemetry, and code structure that
makes proper telemetry hard to add. Output is a ranked list of
findings; observablip never edits files.

## 1. Goal

Reviewing real-world code for observability gaps is high-value but
tedious. The reviewer has to hold three different concerns in mind at
once — *what's not instrumented*, *what's instrumented badly*, and
*what's structured in a way that resists instrumentation* — and apply
them across a codebase the reviewer doesn't fully know. observablip
does that walk in one shot, filters out files where the questions
don't apply, and surfaces a small number of high-signal findings the
user can act on.

The skill targets the user invocation pattern "observablip this
package" / "audit telemetry in `pkg/foo`" / "find missing spans" /
"review o11y on this PR". It is "quick" by design — single mode, lean
SKILL.md, no patches, no fixtures, no executables.

## 2. Non-goals

- Apply patches, autofix, or write code into the target.
- Suggest architectural rewrites beyond the current function/file.
- Review for security, performance, correctness, or test coverage.
- Recommend specific telemetry vendors or backends.
- Operate from artifacts other than source files (no binaries, traces,
  or runtime data).
- Score, grade, or rate the codebase. observablip enumerates issues;
  the user judges magnitude.
- Provide a survey/repo-wide posture mode. (Considered and dropped —
  doxcavate-mimicry without clear demand. Add later if needed.)

## 3. Invocation

Single mode: `audit`. No subcommand routing.

Inputs:

- **Target** — path or glob. Defaults to `.`. Resolved against the
  user's current working directory.
- `--diff <ref>` — optional. Restrict findings to lines changed vs.
  `<ref>` (e.g. `HEAD~1`, `origin/main`). PR-friendly.
- `--lang <id>` — optional override. By default, language is detected
  per file from extension and import patterns.
- `--max <N>` — optional. Output cap (default 20). Overflow is
  surfaced, never silently truncated.

Trigger phrases the `description` field must front-load: "observablip
this", "audit observability", "review telemetry", "find missing
spans", "audit o11y in …", "review my logging".

## 4. Pipeline

Five steps. Each step is mapped to a reference file in section 7.

### 4.1 Resolve target

Expand the path/glob (or `--diff` rev range) to a concrete file set.
Cap file count at a configurable threshold (default 200). Above the
cap, halt with a "narrow the target" prompt — never partially scan.

### 4.2 Per-file precheck

Before any rubric runs, each file is screened for *observable surface
area*. A file passes if it has at least one of:

- A public/exported boundary — HTTP handler, RPC server method, CLI
  entry, queue consumer, scheduled job.
- An external I/O call — network, DB, filesystem, subprocess, or
  channel-spanning operation.
- Non-trivial error edges — at least two distinct error returns or
  raise/throw points.

Files that do not pass are excluded with a one-word reason from a
fixed set: `generated` (matches the `Code generated by … DO NOT EDIT`
header or equivalent), `no-io`, `test-fixture`, `pure-data`,
`vendored`. Excluded files appear only in the footer aggregate; they
never produce findings.

### 4.3 Apply rubric

Three dimensions, each in its own reference file (loaded only when
the agent is about to evaluate that dimension on the surviving file
set):

1. **missing-telemetry** — operations without spans, errors with no
   structured context, missing structured logs at boundaries, missing
   metrics on hot paths, missing trace/request ID propagation.
2. **poor-practices** — log-then-throw, log-as-print, unstructured
   string logs, raw PII/secrets in log args, log spam in tight loops,
   ambient `time.Now()`, error wraps with no metadata.
3. **structure-for-telemetry** — functions mixing unrelated
   operations, no clear request/operation boundary, errors as opaque
   strings instead of typed/categorized, ambient state vs. context
   propagation, deep nesting that hides error edges.

Each rubric reference contains "what to look for" patterns *and* an
explicit "what NOT to flag" anti-pattern set, paired with correct/
wrong code examples (per repo `skills/CLAUDE.md` rule 3.2).

### 4.4 False-positive review pass

Every candidate finding goes through a second look against a
false-positive rubric in `references/false-positive-review.md`. The
FP pass asks, for each candidate:

- Is this a deliberate no-op fastpath where instrumentation would be
  noise?
- Is the missing log/span already covered by an outer middleware,
  decorator, or interceptor visible in the same file?
- Does the surrounding code already handle this concern at a higher
  boundary the candidate didn't see?
- Is the "missing context" actually present elsewhere in the call
  chain?

Findings that survive are kept. Dropped ones are silently discarded;
their count surfaces in the footer (`N candidates dropped by FP
review`).

### 4.5 Rank and emit

Surviving findings are ranked by, in order:

1. Dimension severity. `missing-telemetry` at request boundaries
   outranks `structure-for-telemetry`, which outranks
   `poor-practices`. Severity per finding is `high` / `med` / `low`
   per the rubric file.
2. Blast radius. Findings on a public/exported boundary outrank
   findings inside an internal helper.
3. File path (lexical, for stability).

Output is capped at `--max` (default 20). Overflow is surfaced as
`...and M more, narrow the target` — never silent truncation.

## 5. Output schema

Each finding is a structured Markdown block. The structure is fixed
so a follow-up turn ("now fix #3") can address findings by index.

```text
[3] missing-telemetry · high · pkg/ingest/handler.go:84
    Operation `IngestBatch` runs three external calls (Kafka publish,
    S3 put, Postgres write) without a span or structured log at the
    boundary. A failure in any one is currently indistinguishable
    from the others in production logs.
    Suggest: wrap the operation in a span named `ingest.batch` with
    attributes batch_size, source. Emit one structured log at start
    and one at end with a stable event name.
```

Required fields:

- **index** — 1-based, stable within a single run.
- **dimension** — one of `missing-telemetry`, `poor-practices`,
  `structure-for-telemetry`.
- **severity** — `high` | `med` | `low`.
- **`file:line`** — line numbers point to the most informative line
  for the finding (start of the operation, not start of the file).
- **why this matters** — 1-2 lines, concrete consequence. "A failure
  in any one is currently indistinguishable from the others in
  production logs" is acceptable; "improves observability" is not.
- **suggest** — *shape* of the fix, not a code patch. Names a span,
  metric, log event, or structural change. If the codebase already
  uses OTel / slog / zap / structlog, the suggestion uses that
  vocabulary; otherwise it stays neutral.

Trailing footer (always emitted):

```text
---
scanned: 47 files
skipped: 19 (generated: 8, no-io: 6, test-fixture: 4, pure-data: 1)
fp-dropped: 12
shown: 20 of 23 findings — narrow the target with a path or glob
```

## 6. Hard rules

Each rule has an audit cue mechanical enough for a reviewer (or future
skill-review agent) to check.

1. **Read-only.** observablip never edits files, never executes code
   in the target codebase, and never runs commands inside the target.
   **Audit:** no `Edit` / `Write` / `Bash` tool calls touching paths
   inside the target during a run.
2. **No author attribution.** Findings name behaviors, not authors.
   No `git blame` lookups; no usernames, emails, or handles in
   findings.
   **Audit:** `grep -E '@[a-zA-Z0-9_-]+|<[^>]+@[^>]+>'` over emitted
   findings returns nothing.
3. **No verbatim secrets/PII.** A finding referencing a log call that
   passes a sensitive field describes the *shape* (`a log call passes
   a raw email field`) rather than quoting the line verbatim.
   **Audit:** findings must not contain string literals matching the
   regex set in `references/poor-practices.md` (basic email, JWT,
   bearer-token shapes).
4. **Bounded output.** Default cap 20 findings; overflow always
   surfaces with a count and a "narrow the target" prompt. No silent
   truncation.
   **Audit:** if `len(findings) > --max`, emitted output contains the
   string `narrow the target`.
5. **Skipped files are explained.** Every excluded file is counted in
   the footer aggregate under one of the fixed reason words. No
   silent exclusion.
   **Audit:** `scanned` + `skipped` in the footer equals the
   resolved file count from step 4.1.
6. **No library prescription unless detected.** If the codebase
   already uses OTel / slog / zap / structlog, suggestions use that
   vocabulary. If nothing is detected, suggestions stay neutral
   ("a span", "a structured log") and never push a specific library.
   **Audit:** for each finding, the suggestion's library tokens
   (`otel`, `slog`, `zap`, `structlog`, etc.) must appear somewhere
   in the file set's imports — or the suggestion uses neutral
   vocabulary only.

## 7. Folder layout

Per `skills/CLAUDE.md` rule 1.

```text
skills/observablip/
├── SKILL.md                          # entry, lean, decision-focused
├── README.md                         # human-facing, links to SKILL.md
├── references/
│   ├── precheck.md                   # observable-surface-area rules + skip categories
│   ├── missing-telemetry.md          # rubric dim 1
│   ├── poor-practices.md             # rubric dim 2
│   ├── structure-for-telemetry.md    # rubric dim 3
│   ├── false-positive-review.md      # FP-pass rubric
│   └── output-schema.md              # finding fields, ranking, footer format
└── assets/
    └── finding-template.md           # filled per finding at emit time
```

No `scripts/` — read-only skill, no executables. All file references
one level deep from `SKILL.md`, per repo rule.

## 8. SKILL.md structure

Target ~150-200 lines, well under the 500-line cap from
`skills/CLAUDE.md` rule 3.

Sections, in order:

1. **Frontmatter.**
   - `name: observablip`
   - `description:` what + when + user trigger phrases (the trigger
     list from section 3, front-loaded). Must contain no `<` or `>`
     per the frontmatter-safety rule.
   - `license: MIT`
   - `compatibility: Designed for Claude Code (or similar agentskills.io-compatible clients).`
2. **Install command** — `npx skills add xornivore/skills@observablip --agent claude-code -y`.
3. **When to use** / when not to use.
4. **Hard rules** — the six from section 6, each with its audit cue.
5. **Reference index** — one bullet per reference file, tagged with
   the pipeline step that loads it (precheck → rubrics → FP review
   → rank → emit). References are pulled in at the action step, not
   eagerly in a "see also" preamble.
6. **Top-level action checklist** — the five pipeline steps from
   section 4, each linking to the reference it loads.

Voice: imperative, no hedging. "When invoked, do X" beats
"observablip does X." Per repo rule 3.1.

## 9. Validation

Three gates, in order:

1. **Spec conformance.**
   - `pnpm lint` clean (markdownlint-cli2 over `**/*.md`, with the
     specs dir excluded by config).
   - `npx skills-ref validate ./skills/observablip` exits 0.
2. **Self-audit cues.** Each hard rule in section 6 carries a
   mechanical audit cue. A reviewer can run them by hand without
   re-deriving the test.
3. **Hand-validation example set.** Inline within the relevant rubric
   reference (not a separate fixture dir), 4-6 short code snippets
   covering:
   - HTTP handler with no span — should produce a finding.
   - Pure data struct file — should skip with reason `pure-data`.
   - Generated file with header — should skip with reason `generated`.
   - `log.Info(err); return err` — should produce a finding under
     `poor-practices`.
   - Function mixing two unrelated I/O ops — should produce a
     finding under `structure-for-telemetry`.
   - Already-instrumented handler — should produce zero findings
     (FP pass success).

   These also serve as the correct/wrong examples required by repo
   rule 3.2.

No automated test runner. The skill is prose-and-rubrics; there is
no code to unit-test.

## 10. Repo integration

These belong in the implementation plan, not in the skill itself:

- Add an `observablip` row to the top-level `README.md` skills
  table (per repo rule 4).
- Implementation lands on the existing `skill/observablip` worktree
  branch in `.worktrees/observablip`.
- Conventional-commit format: `feat(observablip): describe the
  skill`. No AI-tool attribution per repo rule 7.
- Pre-commit hook auto-fixes staged Markdown via `lint-staged`. Do
  not bypass with `--no-verify`.

## 11. Open questions

None at spec time. Decisions captured above as defaults; future
tuning (rubric examples, severity thresholds, skip categories) is
expected to happen via normal commits to the references, not by
re-spec.
