# Precheck — observable surface area

Load this reference at pipeline step 4.2. Apply it to every resolved file
before any rubric runs. Files that pass proceed to step 4.3; files that fail
are logged with a skip reason and counted in the footer (see
[output-schema](./output-schema.md)).

## Pass criteria

A file passes the precheck if it has at least one of:

1. **A public/exported boundary** — HTTP handler, RPC server method, CLI entry, queue consumer, scheduled job.
2. **An external I/O call** — network, DB, filesystem, subprocess, or channel-spanning operation.
3. **Non-trivial error edges** — at least two distinct error returns or raise/throw points.

### Criterion 1 — public boundary

A file meets this criterion when it exposes a function or method that a
transport layer, scheduler, or message broker calls directly. Functions
exposed to an HTTP router, gRPC server registration, CLI framework, queue
consumer loop, or cron scheduler count; package-internal helpers reached
only by other package code do not.

**Correct (passes):**

```go
func (s *Server) HandleIngest(w http.ResponseWriter, r *http.Request) error {
    // registered with http.Handle — transport boundary
}
```

A request handler registered with an HTTP router — passes criterion 1.

**Wrong (does not pass):**

```go
func validatePayload(b []byte) error {
    // called only by HandleIngest; never exposed to a transport
}
```

A package-internal helper with no transport exposure — fails criterion 1.

### Criterion 2 — external I/O call

A file meets this criterion when it contains at least one call that crosses
a process boundary: network socket, database driver, filesystem read/write,
subprocess exec, or a channel operation that spans goroutines or OS threads.
Pure in-memory computation and string manipulation do not qualify.

**Correct (passes):**

```python
def fetch_config(bucket: str, key: str) -> dict:
    obj = s3_client.get_object(Bucket=bucket, Key=key)
    return json.load(obj["Body"])
```

An S3 network call crosses the process boundary — passes criterion 2.

**Wrong (does not pass):**

```python
def merge_configs(base: dict, override: dict) -> dict:
    return {**base, **override}
```

Pure in-memory dict merge with no I/O — fails criterion 2.

### Criterion 3 — non-trivial error edges

A file meets this criterion when it contains at least two distinct error
return or raise/throw sites. A function that returns a single fixed error
or panics on one path does not qualify; two independent error paths — from
different operations or conditions — do.

**Correct (passes):**

```go
func LoadConfig(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("read: %w", err) // error edge 1
    }
    var cfg Config
    if err := json.Unmarshal(data, &cfg); err != nil {
        return nil, fmt.Errorf("parse: %w", err) // error edge 2
    }
    return &cfg, nil
}
```

Two distinct error returns from independent operations — passes criterion 3.

**Wrong (does not pass):**

```go
func FormatName(first, last string) string {
    return first + " " + last
}
```

No error return or raise site at all — fails criterion 3.

## Skip reasons

When a file fails all three pass criteria, assign exactly one reason from
this fixed set. Do not invent new reasons; if none fits, default to `no-io`.

### `generated`

Trigger: the file begins with a machine-generated header. Recognized patterns:
`// Code generated by ... DO NOT EDIT` (Go), `# Generated by ...` (Python),
`// @generated` (TypeScript/JavaScript). The header must appear at or near
the top of the file (within the first five non-blank lines).

**Correct (reason applies):**

```go
// Code generated by protoc-gen-go. DO NOT EDIT.
// source: api/ingest.proto

package api
```

Machine-generated header present — assign `generated`.

### `no-io`

Trigger: the file contains application logic (conditionals, loops, function
definitions) but passes none of the three criteria above. Utility functions,
formatters, validators, and similar helpers with no external I/O and no
transport boundary fall here.

**Correct (reason applies):**

```python
def slugify(text: str) -> str:
    return re.sub(r"[^a-z0-9]+", "-", text.lower()).strip("-")
```

Application logic with no I/O and no transport boundary — assign `no-io`.

### `test-fixture`

Trigger: the file lives under a fixture directory or matches a test-fixture
naming convention. Recognized patterns: paths containing `testdata/` or
`__fixtures__/`, filenames ending in `.golden`.

**Correct (reason applies):**

```text
pkg/ingest/testdata/valid_payload.json
```

Path contains `testdata/` — assign `test-fixture`.

### `pure-data`

Trigger: the file declares only types, structs, dataclasses, constants, or
enums, with no methods beyond trivial accessors (getters/setters that read or
write a single field). A file with any method that performs computation,
I/O, or error handling does not qualify.

**Correct (reason applies):**

```python
from dataclasses import dataclass

@dataclass
class IngestRecord:
    id: str
    body: bytes
    source: str
```

Only a dataclass declaration with no methods — assign `pure-data`.

### `vendored`

Trigger: file path falls under `vendor/`, `node_modules/`, or a directory
listed in `.gitignore` as a dependency cache. Path-based; no example needed.

## What NOT to skip

- Files containing only test code (`*_test.go`, `tests/test_*.py`) are NOT
  auto-skipped. Tests can contain o11y anti-patterns the user wants flagged.
  Apply the three pass criteria normally; skip only when all three fail.
- Files smaller than N lines are NOT auto-skipped on length alone. Line
  count is not a signal of observable surface area.

## Action checklist

When the agent reaches step 4.2:

1. For each resolved file, check pass criteria 1–3 in order; the first match
   wins — stop evaluating once a file passes one criterion.
2. If none match, assign a skip reason from the fixed set above. If no reason
   fits, default to `no-io`.
3. Record the file path and reason in the precheck rejection log (consumed by
   the footer at step 4.5 — see [output-schema](./output-schema.md)).
4. Pass surviving files to step 4.3.

**Audit:** every file resolved at step 4.1 must appear exactly once in
either the surviving set or the rejection log.
